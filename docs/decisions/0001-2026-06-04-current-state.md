# 2026-06-04 Current State

## Origin

The app was generated from Bolt prompts asking for a browser-based way to query a Supabase database using a public project URL and API key, even when the user does not own the database.

The follow-up prompt expanded the baseline into a local-first Supabase explorer:

- Persist configuration and settings to local storage.
- Support multiple Supabase connections.
- Show an initial connection setup screen when no databases are configured.
- Let users manually add and remove table names from the UI without implying that database tables are deleted.
- Provide a query screen for the selected table.
- Prefer established libraries over custom query-builder or editor code.
- Publish the app through GitHub Pages.

## Current Architecture

The project is a Vite, React, and TypeScript single-page app. Tailwind CSS handles styling. Supabase access uses `@supabase/supabase-js`. Visual filters use `react-querybuilder`, and advanced query editing uses CodeMirror with SQL syntax support.

The app stores user settings in browser localStorage under `supabase-explorer-settings`. The stored settings include configured connections, each connection's display name, Supabase URL, API key, saved table names, the active connection, and the active table.

Supabase clients are created through a small cache keyed by URL and API key. This avoids recreating clients while allowing users to switch among multiple configured databases.

## Connection And Table Flow

When no connections exist, the app shows a setup screen for Supabase URL, anon/public API key, and optional display name. After a connection is added, the main explorer layout appears.

The sidebar lists connections and saved table names. Users can add table names manually and remove them from the UI with minus-sign actions. Removing a table only removes the saved table entry from localStorage; it does not delete anything from Supabase.

## Query Flow

The selected table opens in the query view. The app fetches rows with a default page size and displays discovered columns based on returned data. The visual filter tab builds Supabase/PostgREST filters from `react-querybuilder` rules. The advanced tab lets users enter a raw query shape.

The intended behavior is that visual filters may query automatically as they change, while advanced query edits should not automatically submit requests. Advanced queries should run only through an explicit submit action such as `Ctrl+Enter`, `Cmd+Enter`, the Run button, or Refresh.

## Deployment

The repository already includes a GitHub Pages workflow at `.github/workflows/deploy.yml`. The workflow runs on pushes to `main`, installs dependencies with `npm ci`, builds with `npm run build`, uploads the `dist` artifact, and deploys it through GitHub Pages.

Vite is configured with `base: './'`, which supports relative asset paths for GitHub Pages hosting.

## Open Issues Captured In This Decision

- The visual rule builder must allow values to be entered and must apply all supported operators consistently.
- Rule translation should be centralized so request execution and request URL previews agree.
- Nested visual rule groups should be supported instead of silently ignored.
- Advanced query typing must not spam Supabase requests.
